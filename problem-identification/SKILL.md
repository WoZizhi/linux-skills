# Linux内核问题定位专家

本skill指导你严谨、安全地完成linux内核问题定位和方案制定的流程。内核代码实现复杂，有时候模块和模块间还有依赖关系（例如nfs调用sunrpc和server通信，文件系统ext4\xfs等基于block层对存储介质nvme/hdd等做IO操作，内存回收也会涉及文件系统操作等），基于问题现象定位触发流程和根因是一个复杂的过程，必须深入模块代码实现逻辑才能从问题现象找到问题根因，而且有时候同一个现象可能有多条触发路径，因此对现象涉及的代码流程要排查彻底，避免遗漏。

核心原则：
+ 不做代码修改和实现，仅分析问题根因和提供修复方案，问题根因和修复方案以markdown文件形式输出
+ 内核代码实现非常复杂，代码量也很大，分析问题根因时要深入理解问题流程对应的内核模块实现。有时候模块之间还有依赖关系，例如nfs和sunrpc，文件系统（ext4\xfs）和block，block和驱动（nvme、scsi）等，如果问题流程涉及跨模块，对应模块的流程也要深入理解到位，确保最终问题定位正确，和代码实现对应得上
+ 不要做假设，不要隐藏困惑，如果用户描述的问题流程或现象和代码实现不匹配，尽管提出疑惑
+ 以代码为准、文档作线索：内核文档（Documentation/）相对代码常有滞后（例如 null_blk.rst 仍写 bio-based 模式，但代码里早删了）。定位以源码为准；文档只用来建立概念/找入口，拿文档去对现象对不上时，要怀疑文档过期而不是怀疑代码，并就近核对代码确认。
+ 修复方案必须可以解决当前问题，并且深入理解修改的子系统，确保修复后不会对当前模块引入新问题，这些问题不光是代码层面的UAF、空指针解引用、内存泄漏等，还包括流程逻辑变更是否会导致逻辑问题，例如当前模块属于磁盘文件系统，修改后会不会导致镜像损坏或文件系统不一致，再例如当前模块属于网络文件系统（nfs、cifs等），修改后会不会导致新的卡死问题（包括网络异常、server重启等异常场景）
+ 修复方案必须完备，要举一反三，当用户没有给定问题根因而只是问题表象后，需要确认是否有多种路径（情况）可能导致相同的现象
+ 修复方案改动尽可能小，不要对核心逻辑有太大影响或变更太多，尽量把改动影响控制到最小范围

## 从栈/崩溃报告落到代码（操作层）
用户给的输入常常不是"描述流程"，而是一份**栈/崩溃报告**（oops/panic/warn/WARN、KASAN/KMSAN/KFENCE/KCSAN 报告、hung task、lockdep splat、softlockup 等）。这份报告是定位的**事实起点**，不是辅助材料——先把报告本身读懂、落到代码坐标，再谈理解模块。不要拿到栈就急着从头读子系统，先用报告把它该指的地方指到。

### 读栈的固定工序
1. **找崩点与访问点**：
   - `RIP` / `IP` / `#PF` 行里要分清"是这条指令自己出错"还是"这条指令访问的地址出错"。对 `#PF`/`#DF`/segfault 类，报告里通常有 `address:` 字段——**真正出问题的常是这个地址，不是 RIP**，根因方向完全不同（前者看指令操作的内存，后者看为什么会跳到/解引用这个地址）。
   - KASAN/KMSAN 报告：标题行（`BUG: KASAN: slab-out-of-bounds in ...`）直接点出缺陷**类型**和**触发的访问点函数**，这是根因候选类型树的第一入口。
2. **读懂 Call Trace 的结构**：
   - 栈是逆序（最近的调用在顶部，越往下越早），注意有的报告是反的，以报告自身的标注（` [<...>] `）为准，别假定方向。
   - 前缀 `?` 的帧是"不确定/可能不是真实调用链"，**可信度低**，不要据此逆推根因。
   - 栈被截断（`...`）或某帧明显跳变/不连续，可能是栈溢出、`CONFIG_FRAME_WARN`、`CONFIG_STACKTRACE` 未开或被破坏——**别当完整栈去逐帧逆推**，要说明"栈不完整，以下推断基于可见帧"。
   - KASAN 的报告通常还附带"access point"和"freed/allocated at"两段调用栈，后者才是**根因栈**——出问题的对象在哪分配/释放的，比访问点栈更接近根因。
3. **把符号+偏移落到 file:line**：
   - 栈里形如 `func+0x8c/0x200`：`0x8c` 是**函数内偏移**（不是指令地址），`0x200` 是函数大小（可用来核对偏移有没有越界）。
   - 实际定位时，用户通常会**直接把出问题的代码发过来**，此时**以用户给的代码为准**，跳过解码这步直接进入读码。
   - 仅当环境具备带调试符号的 `vmlinux` 时才用解码工具，且本环境默认**不编译内核、无 vmlinux**：可选工具有 `scripts/faddr2line vmlinux 'func+0x8c/0x200'`（首选，处理 `func+offset` 格式、能展开 inline 链；addr2line 不吃 KASLR 故不用）、`scripts/decode_stacktrace.sh`（批量解整份栈）、`objdump -d`/`nm`/`readelf`（binutils，crash/gdb 未装时兜底）。无 vmlinux/未开 `CONFIG_DEBUG_INFO` 时只能解到符号名+偏移，解不到行号——这种情况**明确标注"未解码到行号，需用户提供代码或带符号 vmlinux"**，不要静默跳过，更不要凭符号名臆测偏移对应哪一行。

### 根因候选类型树（由栈反向命中哪一类，不是正向扫）
拿到栈和崩点类型后，先按下面这棵树生成**候选根因假设**，而不是一头扎进某个函数读实现。这棵树的类别与 bug-hunting skill 的分类矩阵 A–G 一致（UAF/null-ptr-deref/越界/double-free、资源分配释放顺序/返回值未判/锁未初始化、锁序/内存序/KCSAN竞态、缺调度点softlockup/原子上下文睡眠、错误路径/逻辑、perf/cleanup、UBSAN/KASAN/KMSAN/KCSAN 硬UB）——此处用法相反：bug-hunting 是按分类正向扫代码找缺陷，定位是**由报告反向命中属于哪一类**，可参考该矩阵确保候选覆盖面，不在此重复列细则。
对每个候选，问自己：**"如果根因是它，栈上还该出现什么、不该出现什么？"**——用栈的既存事实去否决候选，而不是用候选去解释栈。能被栈上某帧否决的候选，淘汰；无法否决的留下。

### 确认根因的可证伪门槛
"确认根因无误"不是"想通了"，而是要答得出三件事，否则不算确认、不要进修复方案阶段：
1. **触发条件**：什么输入/并发/错误码/路径会走到这个崩点（不是"可能"，要能指到具体路径）。
2. **整条栈可解释**：Call Trace 上**每一帧为什么会出现**都要能解释，而不只是崩点那一帧。解释不通的帧往往就是根因不在你设想位置的信号——回去重读那一帧对应的代码。
3. **修复能阻断它且不引入新问题**：修复点必须落在"能切断触发路径"的地方，且对当前模块/相邻模块不产生 UAF/死锁/文件系统不一致/网络栈新卡死等副作用（这一条与 implementation skill 的约束一致）。
答不出"为什么这帧会出现"的，多半根因还没找对——这是实战中最有效的否决信号，宁可重读也不要带病进入修复方案阶段。

## 执行流程
1. 读栈/报告落到代码：若用户提供的是栈/崩溃报告（含 `func+offset` 形式符号），先按"从栈/崩溃报告落到代码"操作层读懂报告——分清崩点与访问点、读懂 Call Trace 结构与可信度、按符号+偏移落到 `file:line`。**用户通常会直接把出问题的代码发过来，此时以用户给的代码为准，跳过解码直接进入读码**；仅当有带符号 vmlinux 才用 `scripts/faddr2line`/`decode_stacktrace.sh` 解码。这一步只把报告变成代码坐标，不下根因结论。
2. 问题根因确认： 根据现象或已落到代码的崩点，先用根因候选类型树生成候选根因假设并用栈上事实逐个否决（**由栈反向命中属于哪一类**，不是从头读子系统），对留下来的候选深入理解对应流程的模块实现（含跨模块依赖，如 nfs↔sunrpc、文件系统↔block、block↔驱动），找到合理的问题触发流程（用户也可能提供的不是现象而是触发流程，自己视情况而定），结合代码实现为用户确认问题触发流程和根因。此阶段仅分析问题，不提供修复方案，等用户确认根因无误后进入下一流程。**"确认无误"以"可证伪门槛"三项为准**（触发条件、整条栈可解释、修复能阻断且不引入新问题），答不出就回去重读，不要带病进入下一阶段。
3. 问题修复方案：根据问题根因，提供修复方案，如果有多种方案，请按照改动影响小、实现简单的顺序提供，这个阶段只提供方案，不做代码修改和实现
4. 把问题根因和修复方案输出到markdown文件，markdown文件必须创建在当前工作目录（即linux内核源码根目录，例如`/home/wzz/code-watch/linux`）之下，禁止输出到`/tmp`等临时目录。如需归类可放在源码根目录下的子目录中，但路径必须位于源码根目录之内。
