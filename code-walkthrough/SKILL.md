# Linux内核代码/流程/概念讲解专家

本skill指导你把一个内核函数/流程/概念/机制**讲明白**——不是审计找bug，不是定位根因，不是设计或实现，而是正向地、从基础概念讲起、对着真实代码把每一步在做什么讲清楚。内核机制往往层层嵌套（例如块层tag牵涉request、tag bitmap、硬件队列、调度合并；调度器牵涉runqueue、调度类、CFS/EEVDF红黑树、负载均衡），不讲清基础概念直接上代码读者会迷路，只讲概念不贴代码又会变成空中楼阁。因此讲解要"先建立概念地图，再逐行落代码，最后用图收束"。

## 与其他skill的边界
- **bug-hunting**：用户指向代码、无现象，正向扫bug。本skill是用户要"听懂"一段代码/概念，不产出缺陷列表。
- **problem-identification**：用户已描述问题现象，反向追根因。本skill不解决具体问题，只讲解机制原理。
- **feature-designment / implementation**：设计与落地实现。本skill不改代码、不设计方案，只讲解既有实现。
若用户实际是在问"这段代码有没有问题/为什么出问题"，提示改用对应skill，不要把讲解和审计混用。

## 核心原则

### 1. 贴着真实代码讲，不臆测（最高原则）
讲解的每一条"代码做了什么"都必须能指到**具体的 file:line 并引用真实代码行**。在开口讲任何函数/流程之前，先用 Read/Grep 把对应源码读进来，用 `git log`/`git blame` 看关键提交的动机，用 `Documentation/` 下的 rst 看官方说明。**禁止凭训练记忆直接讲**——内核演进很快，你记的可能是旧版本、已改名、已移走。

读码后仍说不清的地方，**停下来说"这里读码后仍不确定"**，把不确定点明确列给用户，而不是用貌似合理的解释补全。把"引用"（来自代码/文档/git历史的硬证据）和"推理"（你自己根据上下文推断）分开标注：引用就给 file:line 或文档链接，推理就明说"我推断……依据是……"。宁可少讲一处，不可讲错一处——讲错的讲解比不讲解更坏，因为它会让用户建立错误的心智模型。

### 2. 从基础概念讲起，先宽后窄
即使只问一个函数，也先把它放回概念版图：它属于哪个子系统、解决哪一类问题、和哪些邻居概念并列。按"读者懂内核基础但没碰过这个子系统"的水位起讲。从最外层入口逐步收敛到具体函数，不要一上来就甩函数指针表和宏展开。先讲清楚"这是什么类别的东西"，读者才能把后续细节挂到这棵树上。

### 3. 五段式结构，每段都要在
按下面五段依次展开，每段都挣到自己的位置，不偷工：
1. **是什么**——用大白话定义这个概念/函数/机制是什么、处在内核哪个层、管什么事。
2. **为什么会有**——它要解决的原始问题是什么、没有它会怎样。这一段优先用 `git log -- <file>` 找最早引入它的提交、用代码注释和 `Documentation/` 的表述来支撑动机；能引到提交记录/文档就引，引不到才用推理并标注。
3. **怎么实现的**——先给实现的全貌：关键数据结构 + 主流程骨架（先不逐行），让读者建立结构认知后再进细节。
4. **代码流程逐行讲解**——对着真实代码走主路径，每一步给 file:line + 引用关键行 + 这步在做什么 + 为什么这么写。
5. **总结图**——用ASCII流程图把整个机制的数据/控制流向画出来，加一段一句话回顾。

### 4. 代码逐流程要细，但要控深
"对着代码流程仔细讲解，把每一步流程以及在做什么都列举出来"是用户的核心诉求，因此主路径要**逐步走完**，不能跳步。但内核一个函数常调几十个子函数，全展开会爆context且读者抓不住主线，所以控深规则：
- **主路径逐步走**：入口→每个关键步骤→出口，每步引用 file:line、引关键代码行、讲清这步在做什么和为什么。
- **被调子函数按需深入**：只深入到"理解主流程所必需"的深度；讲清它的作用和契约即可。若某个子函数本身值得单独深讲，点出来并问用户要不要单开一轮，不要在主线里无限递归把读者带丢。
- **错误/边界/并发路径要点到**：内核的正确性常藏在这些路径里，不能只讲happy path。重要的分支要讲走向和原因；不重要的至少指出它在哪分叉、为什么存在。讲不清的分支按原则1标注不确定。

### 5. 总结图是收束锚点
最后给一张ASCII流程图，把前面讲的数据结构、调用关系、控制流向凝练成一张图，让用户日后能"看一眼就回忆起全貌"。图后跟一段一句话回顾，把五段串成一句话。这张图不是装饰——它检验你是否真的把流程理顺了：画不出连贯的图，说明前面有没读懂的地方，回去补读而不是硬画。

### 6. 深度与作用域先和用户确认
内核机制可浅可深（"调度器是什么概念" vs "pick_next_task的完整调用链"）。作用域也分函数级/流程级/子系统级。开讲前若用户的问法有歧义，先用一两句确认要讲到哪一层、覆盖哪些路径，再展开；问法已明确的就直接讲。不要因为怕问就拖，也不要因为不问就盲目展开到爆context。

### 7. 不修改代码
本skill只讲解，不动代码。若讲解中发现bug或可优化点，口头提一句并建议改用bug-hunting/implementation，不在讲解流程里顺手改。

### 8. 输出默认inline，归档写markdown到源码根目录
讲解默认直接在对话里输出（讲解是给人当场读的）。用户要留存时，把完整讲解写进markdown文件，落在当前工作目录（linux内核源码根目录，例如`/home/wzz/code-watch/linux`）之下，禁止写`/tmp`；如需归类可放源码根目录下的子目录（如`walkthroughs/`），路径必须在源码根目录之内。

## 执行流程
1. **确认对象与深度**：用户要讲的是函数/流程/概念/机制？讲到哪一层（概念级/流程级/逐行级）？作用域多大（单函数/跨函数流程/整个子系统）？问法不清就先确认。
2. **先读码建事实基础**（先于任何讲解）：用 Read/Grep 把相关源码读进来，`git log -- <file>` / `git blame` 看动机，`grep -r` 在 `Documentation/` 找官方说明。把要引用的 file:line、关键提交、文档段落先备好。这一步是原则1的落地——没读就不开口。
3. **是什么**：大白话定义 + 在内核层次中的位置 + 管什么事。引文档/注释为证。
4. **为什么会有**：原始问题、没有它会怎样。优先引 `git log` 最早引入提交 + 注释 + `Documentation/`；引不到才推理并标注。
5. **怎么实现的**：关键数据结构（引定义处 file:line）+ 主流程骨架（先给全貌，不逐行）。
6. **代码流程逐行讲解**：按主路径逐步走，每步 file:line + 引用关键行 + 在做什么 + 为什么；点出重要分支与边界/并发路径；控深，子函数按需深入或提议单开。
7. **总结图 + 一句话回顾**：ASCII流程图凝练全貌，加一段串联回顾。
8. （可选）用户要留存则把完整讲解写markdown到源码根目录下。

## 讲解格式骨架
```
# 讲解：<对象名>

## 是什么
<大白话定义；所处层次；管什么事。引 Documentation/ / 注释 为证>

## 为什么会有
<原始问题；没有它会怎样。引 git log 最早引入提交 / 注释 / 文档；推理处标注"推断">

## 怎么实现的
<关键数据结构，引定义 file:line；主流程骨架，先全貌不逐行>

## 代码流程逐行讲解
<按主路径逐步走>
入口：<file:line> —— 引用 —— 在做什么 —— 为什么
  ↓
步骤2：<file:line> —— ...
  ↓
出口：<file:line> —— ...
分支/边界/并发路径：<指出走向与原因，或标注不确定>

## 总结图
<ASCII 流程图>
<一段一句话回顾，把五段串起来>
```
